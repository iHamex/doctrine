# Primary Keys & Generated Values

Every entity in Doctrine must have a primary key, which uniquely identifies it. Doctrine provides a flexible and powerful system for managing these keys, from simple auto-incrementing integers to complex composite keys and universally unique identifiers (UUIDs).

## Identifier Strategies: How IDs Are Generated

The `#[ORM\GeneratedValue]` attribute tells Doctrine how the primary key for an entity should be generated. Choosing the right strategy is important for application performance and scalability.

| Strategy   | How it Works                                                                       | When to Use                                                                | PHP-Side ID Availability |
| ---------- | ---------------------------------------------------------------------------------- | -------------------------------------------------------------------------- | ------------------------ |
| `IDENTITY` | The database assigns the ID on `INSERT` (e.g., `AUTO_INCREMENT`).                  | **Default for MySQL/MariaDB, SQLite.** Simple and efficient for most cases. | After `flush()`          |
| `SEQUENCE` | Doctrine fetches IDs from a database sequence *before* `INSERT`.                     | **Default for PostgreSQL, Oracle.** Allows for pre-`flush` ID access.    | Before `flush()`         |
| `UUID`     | A universally unique identifier is generated by Doctrine in PHP.                     | Public-facing IDs, distributed systems, hiding record counts.              | Before `flush()`         |
| `NONE`     | You are responsible for setting the ID manually.                                     | Natural keys (e.g., an `email` as a PK) or keys from an external system. | Before `flush()`         |

!!! tip "`AUTO` Strategy"
    If you don't specify a strategy (`#[ORM\GeneratedValue]`), Doctrine defaults to `AUTO`. This tells Doctrine to pick the best strategy for the current database platform (`IDENTITY` for MySQL, `SEQUENCE` for PostgreSQL, etc.). This is the recommended default for most applications.

---

### `IDENTITY`: Simple Auto-Increment

This is the most common strategy. The database is responsible for generating the ID when a new record is inserted.

```php
#[ORM\Entity]
class Product
{
    #[ORM\Id]
    #[ORM\Column(type: 'integer')]
    #[ORM\GeneratedValue(strategy: 'IDENTITY')]
    private ?int $id = null;
}

$product = new Product();
// $product->getId() is null here
$entityManager->persist($product);
$entityManager->flush();
// $product->getId() is now populated (e.g., 123)
```

**Key Characteristic**: The ID is only available *after* the `INSERT` operation has been flushed to the database.

### `SEQUENCE`: Pre-fetching IDs

This strategy is common in PostgreSQL and Oracle. Doctrine asks the database for a block of IDs from a sequence *before* it needs them, which can improve performance for batch inserts.

```php
#[ORM\Entity]
class Order
{
    #[ORM\Id]
    #[ORM\Column(type: 'integer')]
    #[ORM\GeneratedValue(strategy: 'SEQUENCE')]
    #[ORM\SequenceGenerator(sequenceName: 'order_id_seq', allocationSize: 10)]
    private ?int $id = null;
}

$order = new Order();
// $order->getId() is null here
$entityManager->persist($order);
// $order->getId() is now populated (e.g., 501) BEFORE the flush!
$entityManager->flush();
```
**Key Characteristic**: The ID is available in PHP *before* the `flush()` call. The `allocationSize` is a performance tuning option that determines how many IDs Doctrine fetches at once.

### `UUID`: Globally Unique Identifiers

UUIDs are 128-bit identifiers that are statistically guaranteed to be unique. Doctrine generates them in PHP, meaning they are available immediately.

```php
use Symfony\Component\Uid\Uuid;

#[ORM\Entity]
class Article
{
    #[ORM\Id]
    #[ORM\Column(type: 'uuid', unique: true)]
    #[ORM\GeneratedValue(strategy: 'UUID')]
    private ?Uuid $id = null;
}

$article = new Article();
// $article->getId() is now a Uuid object, e.g., "a1b2c3d4-..."
$entityManager->persist($article);
$entityManager->flush();
```
!!! tip "Use `symfony/uid`"
    For a better developer experience, use the `symfony/uid` component and map your UUID column to the `uuid` type. This gives you a rich `Uuid` object instead of a plain string.

### `NONE`: Manually Assigned Identifiers

This strategy tells Doctrine that you will handle setting the primary key yourself. This is common for entities that have a "natural" primary key.

```php
#[ORM\Entity]
class User
{
    #[ORM\Id]
    #[ORM\Column(type: 'string', length: 180, unique: true)]
    #[ORM\GeneratedValue(strategy: 'NONE')]
    private string $email;

    public function __construct(string $email)
    {
        $this->email = $email;
    }
}

$user = new User('test@example.com');
// The ID ('test@example.com') is already set.
$entityManager->persist($user);
$entityManager->flush();
```

!!! warning "Natural Keys Trade-offs"
    While natural keys can be convenient, they can also cause issues. For example, if a user needs to change their email address, you would have to update the primary key, which can be complex and cascade through many related tables. Using a separate, immutable ID (like an auto-incrementing integer or UUID) is often a more robust design.

## Composite Primary Keys

Doctrine also supports composite primary keys, where the identity of an entity is defined by two or more columns. To define a composite key, you place the `#[ORM\Id]` attribute on each property that is part of the key.

**Important**: Entities with composite keys must use the `NONE` generation strategy. You are always responsible for setting the values of a composite key.

```php
#[ORM\Entity]
class CartItem
{
    #[ORM\Id]
    #[ORM\ManyToOne(targetEntity: Cart::class)]
    private Cart $cart;

    #[ORM\Id]
    #[ORM\ManyToOne(targetEntity: Product::class)]
    private Product $product;

    #[ORM\Column(type: 'integer')]
    private int $quantity;
}

// Creating a new composite key instance
$cartItem = new CartItem();
$cartItem->setCart($someCart);
$cartItem->setProduct($someProduct);
$cartItem->setQuantity(5);
$entityManager->persist($cartItem);
$entityManager->flush();
```

## Database-Generated Timestamps

You can also leverage the database to generate non-primary-key values, such as `createdAt` and `updatedAt` timestamps. This is typically done using database-level defaults.

```php
#[ORM\Entity]
class LogEntry
{
    // ... id
    
    #[ORM\Column(
        type: 'datetime_immutable',
        options: ['default' => 'CURRENT_TIMESTAMP']
    )]
    private \DateTimeImmutable $createdAt;
    
    public function __construct()
    {
        $this->createdAt = new \DateTimeImmutable();
    }
}
```

By setting both a PHP-side default (in the constructor) and a database-side default (in the `options` attribute), you get the best of both worlds:
- Your PHP objects are always in a valid state, even before being persisted.
- The database guarantees the `createdAt` column will be set, even if the data comes from a non-Doctrine source.
